# PRD: 숫자 범위 하드 필터 (가격·순도·분자량·보관온도)

Status: done

## Problem Statement

임베딩 검색은 유형·용도·재질·브랜드 같은 **어휘/의미 신호**엔 강하지만(브랜드도 임베딩이 잘
처리함 — 실측 확인), **숫자 범위 제약**엔 무력하다. "3000만원 이하 제품"은 가격이 검색 색인에
없어 필터되지 않고, 숫자 "3000만"이 어휘로 섞여 "3000 mL" 상품에 오매칭된다. "순도 99% 이상",
"분자량 100 이하", "냉장(2~8℃) 보관" 같은 질의도 마찬가지다.

## Solution

임베딩이 못 하는 **숫자 범위 비교만** 하드 필터로 붙인다. 적재 시 상품의 숫자 속성
(가격·순도·분자량·보관온도)을 kg_embedding에 숫자 컬럼으로 색인하고, 검색 시 SQL `WHERE`로
시맨틱과 **동시에** 건다(메타데이터 필터 벡터 검색). 질의의 제약은 분석기 LLM이 뽑는다.
브랜드·재질 등 범주형은 **임베딩이 이미 처리**하므로 하드 필터에 넣지 않는다.

## User Stories

1. 사용자로서, "3000만원 이하 원심분리기"에 실제로 3천만원 이하 원심분리기만 나오길 원한다. 예산 초과 상품에 시간 낭비 안 하도록.
2. 사용자로서, "순도 99% 이상 시약"에 고순도만 나오길 원한다. 스펙 기준으로 거르도록.
3. 사용자로서, "냉장 보관 제품"과 "2~8도 보관 제품"이 똑같이 작동하길 원한다. 표현이 달라도 같은 의도니까.
4. 사용자로서, 조건에 맞는 게 없으면 엉뚱한 상품 대신 "그 조건엔 없다"고 정직히 듣고 싶다. 가격/스펙 제약을 존중받도록.
5. 사용자로서, 예산을 말하면 그 예산을 절대 초과하지 않길 원한다(자동 완화 없음). 하드 제약이 지켜지도록.
6. 사용자로서, "SIGMA 냉장 시약"처럼 브랜드+숫자제약을 섞어도 되길 원한다. 브랜드는 의미로, 온도는 필터로 처리되도록.
7. 운영자로서, 필터 컬럼을 1줄로 추가·확장하고 싶다. 녹는점·인화점 등을 같은 기계로 붙이도록.
8. 운영자로서, 소스에서 가격·스펙이 바뀌면 필터도 갱신되길 원한다. content-hash로 재적재되도록.
9. 개발자로서, 파서·필터 검색·제약 추출·필터-범인 감지를 실제 LLM 없이 결정적으로 테스트하고 싶다.
10. 사용자로서, 숫자 제약이 없는 질의는 기존과 똑같이 동작하길 원한다(회귀 없음).

## Implementation Decisions

- **아키텍처: 메타데이터 필터 벡터 검색**. 적재 시 숫자 속성을 kg_embedding 숫자 컬럼에 색인하고,
  검색 SQL에 `WHERE ... ORDER BY embedding<->q`로 필터+시맨틱 동시 적용. 사후 하이드레이션
  필터가 아니라 **필터된 집합에서 시맨틱 순위**여야 정확하기 때문.
- **하드 필터 = 숫자 범위 전용**. 임베딩이 못 하는 것만: 가격·순도·분자량·보관온도(+후속
  녹는점·인화점·밀도). 브랜드·재질·유형·손잡이·단위·위험분류 등 **범주/어휘형은 임베딩·키워드가
  처리**(실측: "ATAGO 굴절계"→ATAGO만, "Nalgene 병"→Nalgene만). 하드 필터에서 제외.
- **선언적 필터 레지스트리**: 각 필터를 (이름, source 추출, parse 함수)로 선언. 적재·색인·질의·
  제약추출이 이 레지스트리 위에서 일반화. 컬럼 추가 = 항목 1줄.
- **저장 = 타입 있는 min/max 컬럼**: 필터마다 `_min`·`_max`(원본이 범위면 그대로: "2-8C"→(2,8);
  point면 min=max: 분자량 71.08→(71.08,71.08); 가격은 변형 io_price의 min/max). JSONB 아님(숫자
  비교·인덱스가 깔끔).
- **정규화**: 질의 시 = **분석기 LLM**이 레지스트리 스키마를 받아 제약 추출("3000만원 이하"→
  `price_max=30000000`, "순도 99% 이상"→`purity_min=99`, "냉장/2~8도"→`storage_temp={min:2,max:8}`).
  적재 시 = **결정적 parse 함수**(가격=변형 min/max; 나머지=필드 텍스트에서 숫자 추출; storage는
  소어휘 매핑 "room"→(15,25)). storage 온도매핑(냉장=2~8·냉동≤-20·실온=15~25)은 분석기 프롬프트와
  적재 파서가 **같은 정의** 공유.
- **매칭 = 겹침(overlap)**: 이하(≤X)→`col_min≤X`; 이상(≥X)→`col_max≥X`; 범위[A,B]→`col_min≤B AND
  col_max≥A`. 값 NULL이면 그 필터에서 **제외**(정보 없는 상품 억지 포함 안 함).
- **필터 × 반복 루프**: 필터는 **하드**(자동 완화 없음)이고 루프 내내 **유지**(재정식화는 검색어만
  바꿈). 필터 걸고 0건이면 **필터 빼고 1회 더 검색** → 결과 있으면 = **필터가 범인** → 검색어
  재시도 대신 **구체적 안내**("그 조건에 맞는 상품을 찾지 못했습니다"); 빼도 0이면 검색어 재정식화.
- **분석기 확장**: `Analysis`에 `filters` 추가. `analyze`가 검색어와 함께 필터를 냄. 제약 없으면
  `filters={}` → 기존과 동일(회귀 없음).
- **적재/신선도**: 필터 값은 doc(변형 가격·brand·field_info)에서 나오고 이미 content_hash에
  반영됨 → sync로 갱신. 단 신규 컬럼은 기존 행에 NULL이라 **재적재로 채워야** 함(배포 시 재적재).

## Testing Decisions

- 좋은 테스트는 외부 행동을 공개 인터페이스로 검증(구현 내부 아님).
- **파서**(레지스트리): "2-8C"→(2,8), ">=99.5% (GC)"→(99.5,None or 99.5), "82-86 C"→(82,86),
  "71.08"→(71.08,71.08), "room temperature"→(15,25). 순수 함수, 결정적.
- **필터 추출**(FilterExtractor): 페이크 doc → {price:(min,max), storage_temp:(min,max), …}.
- **필터 검색**(EmbeddingStore.search/keyword_search + filters): 픽스처 행으로 겹침 매칭·NULL 제외·
  다중 필터 AND. 프리아트: 기존 test_ingest_embedding(실 DB 픽스처).
- **분석기**: ScriptedChatModel로 필터 JSON 파싱("3000만원 이하"→price_max), 제약 없으면 빈 필터.
- **필터-범인 감지**(RagRecommender): FakeRetriever가 필터 시 0·무필터 시 >0 → 구체적 안내 방출.
  프리아트: test_rag_recommender(FakeRetriever·ScriptedChatModel).

## Out of Scope

- 범주형 하드 필터(brand·packaging_uom·hazard_type) — 임베딩이 처리. 브랜드 별칭맵(SIAL→Sigma)도 후속.
- 희소 숫자 필터(녹는점·인화점·밀도, 채움율 1~19%) — 레지스트리에 넣기는 쉬우나 v1 TDD는
  가격·순도·분자량·보관온도 4종에 집중(파서 다양성 대표). 나머지는 후속으로 1줄 추가.
- 필터 자동 완화·소프트 필터(랭킹 가중). 하드만.
- prod 배포·재빌드(시연 중, 건드리지 않음).

## Further Notes

- ADR-0018: "하드 필터는 임베딩이 못 하는 숫자 범위 비교에만; 범주형은 임베딩" 결정과 근거(브랜드
  실측)·겹침 규칙·필터-범인 감지·재적재 필요를 남긴다.
- 배포 절차: 신규 컬럼 채우려면 `embed_products --reset`(또는 필터 backfill) 재적재 필요. dev는 3k 재적재.
- 검증: "3000만원 이하 원심분리기"·"냉장 2~8도 시약"이 실제로 필터되는지 라이브 확인.
