# PRD: Route C — 적재 시 LLM 설명으로 상품 임베딩 강화

Status: ready-for-agent
Labels: ready-for-agent

## Problem Statement

현재 상품 임베딩 텍스트는 **상품명 + 속성 값**뿐이다(얇음). 카탈로그 상품명이 대부분 영어라, 한국어 질의가 임베딩 공간에서 상품과 안 붙는다 — "플라스크"가 "Flask"를 못 찾는 KO↔EN 미스매치. 이를 메우려고 **매 채팅마다 LLM 질의이해(QueryAnalyzer) 1콜**로 번역·키워드를 뽑아 검색을 살리고 있다. 즉 인덱스가 얇아서 생긴 문제를 **질의당 반복 비용**으로 땜질하는 구조다.

프로토타입 검증(400 상품, 한국어 질의 15개 recall@5):

| 방식 | 채팅당 LLM | recall@5 |
|---|---|---|
| 현재 인덱스 + 분석기 없음 | 1(생성) | 2/15 |
| **강화 인덱스 + 분석기 없음** | **1(생성)** | **10/15** |
| 현재 인덱스 + 분석기(운영) | 2 | 12/15 |
| **강화 인덱스 + 분석기** | 2 | **13/15** |

## Solution

embed-chat(GraphRAG)의 원리를 이식한다 — **지능을 질의 시점이 아니라 적재 시점으로 앞당긴다.** 적재 시 상품마다 LLM이 **한/영 설명 + 검색 키워드**를 생성해 임베딩 텍스트에 포함한다. 그러면:

- **검색 품질↑**: 강화 인덱스가 어느 조합에서든 검색을 개선(분석기 유지 시 12→13, 분석기 없이 2→10).
- **채팅 비용↓ 여지**: 강화만으로 분석기 없이도 운영 근접(10 vs 12) → 질의당 LLM 2→1로 줄일 선택지가 생긴다.
- **비용은 1회성 적재로 이동**: content-hash 게이팅으로 변경분만 재생성, 매 채팅에서 분할상환.

이 PRD의 범위는 **적재 파이프라인의 강화**다. 질의 경로에서 분석기를 실제로 뺄지(비용↓ vs 품질)는 eval A/B로 판정하는 후속 토글로 둔다.

## User Stories

1. 연구원으로서, "플라스크"·"인듐"처럼 한국어로 검색해도 영어 상품이 잘 나오길 원한다. 번역해 치지 않아도 되도록.
2. 연구원으로서, 서술형·용도 질의("액체를 옮기는 도구")가 더 잘 걸리길 원한다. 상품 설명이 임베딩에 담기도록.
3. 운영자로서, 매 채팅의 LLM 비용을 줄이고 싶다. 검색 지능을 적재 1회로 옮겨 분할상환하도록.
4. 운영자로서, 강화를 켜고 끌 수 있길 원한다(`.env`). LLM 적재 비용을 감당 못 하는 배포는 현행 유지하도록.
5. 운영자로서, 전체 카탈로그(수십만) 최초 강화를 저렴·안정적으로 하고 싶다. Batch API로 절반 비용·대량 처리하도록.
6. 운영자로서, 이후엔 신규·변경된 상품만 재강화되길 원한다. content-hash로 중복 LLM 호출을 막도록.
7. 개발자로서, 설명 생성이 임베딩과 분리·캐시되길 원한다. 모델 교체 시 재임베딩만 하고 재생성은 안 하도록.
8. 개발자로서, 설명 생성 실패가 적재를 막지 않길 원한다. 실패 시 현행(name+값) 임베딩으로 폴백하도록.
9. 개발자로서, 설명 생성을 fake LLM으로 결정적 테스트하고 싶다. 프롬프트·조립·게이팅을 검증하도록.
10. 운영자로서, 강화가 실제 추천 품질을 올리는지 판정관 채점으로 확인하고 싶다(eval A/B). recall이 아니라 최종 품질로.
11. 개발자로서, 강화 임베딩과 현행 임베딩을 A/B 하는 동안 공존시키고 싶다. 별도 테이블/플래그로 안전하게 비교하도록.
12. 운영자로서, 설명 모델을 지정하고 싶다(`.env`). 비용·품질을 조절하도록.
13. 연구원으로서, 강화 후에도 추천 카드의 그라운딩·가격·URL이 그대로이길 원한다(임베딩은 검색용, 근거는 결정적 부착 유지).

## Implementation Decisions

### 강화 파이프라인 (딥모듈)

- **DescriptionGenerator(딥모듈, LLM)**: `describe(name, attributes) → 설명 텍스트`. 상품명 + 속성으로 **한/영 한 줄 설명 + 검색 키워드**를 생성한다(간결·군더더기 없음). 모델은 `RAG_DESCRIPTION_MODEL`(기본 `OPENAI_MODEL`=gpt-4o-mini). LLM 경계 주입으로 fake 테스트 가능. 실측: 상품당 입력 ~105 tok · 출력 ~82 tok.
- **강화 임베딩 텍스트**: `상품명 + 속성 값 + "\n" + 설명`. 실측 ~130 tok/상품.
- **DescriptionStore(캐시)**: `source_id → (content_hash, 설명)`. 설명 생성은 **content-hash 게이팅** — 상품의 이름·속성이 안 바뀌면 재생성하지 않는다. 설명을 **저장**해 임베딩과 분리한다(모델 교체 시 재임베딩만, 재생성 없음). 저장 위치는 임베딩 저장소 옆 별도 테이블/컬럼.
- **폴백**: 설명 생성 실패(오류·타임아웃) 시 현행 `name + 값`으로 임베딩(적재 차단 금지). 기존 임베딩 실패 격리 패턴 계승.

### 적재 통합 (IngestRunner)

- `IngestRunner.apply`가 `set_attributes` 뒤, 임베딩 전에 **강화 켜짐(`RAG_ENRICH`)이면** DescriptionGenerator로 설명을 얻어(게이팅·캐시) 강화 텍스트로 임베딩한다. 꺼짐이면 현행 그대로.
- content-hash 게이팅은 설명·임베딩 둘 다에 적용 — unchanged 상품은 재생성·재임베딩 생략.

### 최초 대량 강화 (Batch API)

- **backfill 명령**(예: `enrich_products`): 강화가 필요한 상품(신규·변경·미강화)을 모아 **OpenAI Batch API**로 설명을 제출한다(비동기·24h·**50% 비용**). 제출 → 폴링 → 완료 시 설명 저장 + 재임베딩. 카탈로그가 작으면 `--sync`로 동시성 기반 즉시 처리도 지원.
- 전체 628,536개 실측 외삽: 표준 API **≈ $42** / Batch API **≈ $21** (1회성). 이후 운영은 변경분만(월 소액).
- 델타 동기화(`sync_poll`)는 저볼륨이라 per-product 표준 API로 강화한다.

### 설정

- `RAG_ENRICH`(0/1, 기본 0 — 하위호환): 적재 강화 on/off.
- `RAG_DESCRIPTION_MODEL`(기본 OPENAI_MODEL): 설명 생성 모델.
- 기존 `EMBEDDING_MODEL`·content-hash 게이팅 재사용.

### 질의 경로 (범위 밖·후속 토글)

- 강화 후 분석기를 **유지**(품질 최대, 13/15, 채팅당 2 LLM)할지 **제거**(비용 최소, 10/15, 채팅당 1 LLM)할지는 `.env` 토글로 두고 **eval A/B로 판정**한다. 이 PRD는 적재 강화까지가 범위이며, 분석기 제거는 A/B 통과 시 별도로 확정한다.

### 검증 (eval A/B, 판정관 채점)

- eval 하네스로 **강화 인덱스 vs 현행 인덱스**를, **분석기 유무 조합**으로 판정관(gpt-4o) 채점한다(recall이 아니라 최종 추천 품질). 강화 임베딩은 **별도 테이블**(예: `kg_embedding_rich` / eval는 `eval_embedding_rich`)에 두어 현행과 공존·비교한다. 프로토타입 recall(2→10, 12→13)의 판정관 확증.

## Testing Decisions

좋은 테스트는 공개 인터페이스로 외부 행동을 검증하고 구현에 결합하지 않는다. 결정적 파트는 fake LLM/counting double로 테스트(기존 `test_ingest_embedding`·`test_embedding_store`·ContextTrimmer counting 패턴 계승).

- **DescriptionGenerator**(fake LLM): 상품명·속성을 프롬프트에 담아 호출하고 반환 설명을 그대로 돌려준다; 실패 시 폴백(빈/예외 → 현행 텍스트).
- **강화 임베딩 텍스트 조립**: name + 값 + 설명이 합쳐진다.
- **content-hash 게이팅**(counting double): 같은 상품 재적재 시 설명 LLM이 **재호출되지 않는다**(캐시 히트). 변경 시 재생성.
- **IngestRunner 통합**: `RAG_ENRICH` on이면 강화 텍스트로 임베딩, off면 현행. 설명 실패해도 적재 성공(폴백).
- **Batch 명령**: 프롬프트 빌드·결과 파싱을 결정적으로 테스트(Batch 제출은 mock). 대상 선별(미강화/변경분)만 포함.
- **eval A/B**: 판정관 채점으로 강화 vs 현행(±분석기) 최종 품질 확증 — 채택 근거.

## Out of Scope

- 질의 경로에서 분석기 실제 제거 — 후속 토글, A/B 통과 시 확정.
- embed-chat 완전판(문서 엔티티 추출 + 관계 + 커뮤니티 요약) — 상품 카탈로그엔 설명 1콜로 충분(과잉).
- 멀티벡터/필드 분리 임베딩(희석 회피 고급안) — 후속 실험.
- 강한 다국어 임베딩 모델 교체(BGE-M3 등) — 별도 A/B(Route B).
- Self-RAG 폴백(근거 부족 시 원문/필드 재검색) — 별도 PRD.
- 이미지/vision 근거(ADR-0010 범위 밖).

## Further Notes

- 근거: 실측 recall A/B(THIN 2 → RICH 10, 분석기 포함 12 → 13) + 토큰 실측(105/82/130)·비용 외삽(~$21 Batch / ~$42 표준, 628k 1회성). 프로토타입 `kg_embedding_rich`(400개)로 확인됨.
- 이 전환은 ADR-0012(임베딩)·0014(RAG 읽기 경로)를 **확장**한다 — 임베딩 텍스트를 풍부화하고, 강화가 통하면 질의이해 비용을 낮춘다. 되돌리기 비용·트레이드오프(적재 LLM 비용↔채팅 절감)가 있으므로 **짧은 ADR 1건**으로 결정을 남긴다.
- 원칙 유지: 임베딩은 **검색 리콜용**이고, 추천 카드의 그라운딩·가격·URL은 계속 **결정적으로** 부착한다(LLM이 상품 사실을 지어내지 않음, ADR-0001).
- 저사양·비용 전제: `RAG_ENRICH` 기본 off로 하위호환, Batch API로 대량 강화, content-hash로 지속 비용 최소.
