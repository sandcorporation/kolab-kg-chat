# PRD: 반복 검색 루프 + 질의생성 복구 + 팔로업 라우팅

Status: done

## Problem Statement

RAG 읽기 경로(ADR-0014/0015)는 비용을 줄이려 **질의생성 LLM(QueryAnalyzer)을 제거**하고 raw
질의로 한 번만 검색·선택한다. 그래서 검색이 첫 시도에서 실패하면(후보에 맞는 상품이 없어
`선택: 없음`) 그걸로 끝나 버린다 — 다른 검색어로 다시 시도하지 못한다. 또 강화 임베딩에도
불구하고 어려운 질의는 한 번의 검색어로 못 잡을 수 있다. 반대로 "그 중 첫 번째 어디 써?"
같은 팔로업 질의에도 매번 새로 검색해 엉뚱한 결과를 낸다.

## Solution

채팅 에이전트를 **에이전틱 반복 검색**으로 바꾼다. 질의생성 LLM을 복구해 원 질의에서 한/영
검색어를 뽑고, **만족스러운 결과가 나올 때까지 최대 N회** 검색어를 바꿔가며 재시도한다.
만족 판정은 기존 선택 단계를 재사용한다(`선택: 없음`=불만족 → 재검색). 팔로업 질의는
검색을 건너뛰고 대화 맥락으로 답한다. 비용(채팅당 LLM 콜)이 늘지만 어려운 질의의 검색
품질이 오른다. 비용 최적화(ADR-0015)를 품질 위해 되돌리는 **새 기본값**이다.

## User Stories

1. 최종 사용자로서, 한 번의 검색으로 안 나와도 시스템이 검색어를 바꿔 다시 찾길 원한다. 첫 시도 실패로 빈손이 되지 않도록.
2. 최종 사용자로서, 한국어로 물어도 영어 상품명을 찾길 원한다. 질의생성이 한/영 검색어를 뽑아주도록.
3. 최종 사용자로서, "그 중 첫 번째 어디 써?" 같은 후속 질문엔 다시 검색하지 말고 방금 얘기한 상품으로 답하길 원한다. 맥락이 끊기지 않도록.
4. 최종 사용자로서, 재시도 중임을 화면에서 알고 싶다. 멈춘 게 아니라 더 찾고 있다는 걸 알도록.
5. 최종 사용자로서, 아무리 찾아도 없으면 억지 추천 대신 정직하게 "못 찾았다"고 듣고 싶다. 잘못된 추천으로 헷갈리지 않도록.
6. 운영자로서, 재시도 횟수 상한을 설정으로 조절하고 싶다. 비용과 품질을 상황에 맞게 balancing 하도록.
7. 운영자로서, 무한/헛도는 재시도가 없길 원한다. 같은 검색어로 반복하면 조기 종료되도록.
8. 운영자로서, 이 변경이 시연 중인 prod에 영향 없길 원한다. prod를 재배포하지 않는 한 그대로 돌도록.
9. 개발자로서, 반복 루프의 각 분기(만족·재시도·소진·무진전·팔로업)를 실제 LLM 없이 결정적으로 테스트하고 싶다. CI에서 안정적으로 검증하려고.
10. 최종 사용자로서, 최종 추천 근거는 여전히 실시간 토큰 스트리밍으로 보고 싶다. 응답이 살아있는 느낌을 유지하려고.
11. 최종 사용자로서, 재시도가 실패한 이전 후보를 반복하지 않길 원한다. 재검색어가 거부된 결과를 학습하도록.
12. 운영자로서, 이 변경이 되돌리기 쉽길 원한다(설정으로 N=1이면 사실상 단일 패스). 문제 시 빠르게 완화하도록.
13. 최종 사용자로서, 추천 카드의 이름·가격·속성은 여전히 소스에서 정확히 온다고 신뢰하고 싶다(환각 없음, ADR-0001/0016).

## Implementation Decisions

- **QueryAnalyzer 복구·확장**(딥모듈): 옛 QueryAnalyzer(한/영 키워드 + 시맨틱 질의를 JSON으로)를
  되살리되 두 능력을 갖춘다:
  - `analyze(query, history)` → **라우팅 + 초기 검색어**. LLM이 `팔로업`인지 판정(히스토리가
    있고 명백히 이전 상품을 가리킬 때만; 첫 턴·애매하면 새 검색 — 보수적)하고, 새 검색이면
    한/영 키워드 + 시맨틱 질의를 낸다. 파싱 실패는 원 질의로 폴백.
  - `reformulate(query, prev_terms, rejected_names)` → **재검색어**. 원 질의 + 직전 검색어 +
    거부된 후보 이름을 보고 **직전과 다른** 검색어를 생성.
- **HybridRetriever.retrieve 시그니처**: raw 질의 대신 **(키워드 목록, 시맨틱 질의)** 를 받아
  키워드마다 keyword_search + 시맨틱 검색 1회 → 합집합·중복제거·top-K + 설명 부착(C 유지).
- **RagRecommender 반복 루프**: `astream`이 다음을 한다(프로토타입 흐름):
  ```
  route = analyze(query, history)
  if route == FOLLOWUP: 히스토리로 답변 스트리밍 → result([]); return
  keywords, semantic = route.terms
  for i in range(N):                       # AGENT_MAX_ITERS(기본 3)
      status(i==0 ? "검색 중…" : "다른 검색어로 다시 찾는 중… i/N")
      candidates = retrieve(keywords, semantic)        # 교체(fresh)
      선택 스트림 → 첫 줄 '선택:' 엿보기:
          만족(유효 선택)  : 나머지 근거 토큰 스트리밍 + result(ids) → return
          불만족 & 마지막  : 정직한 폴백 + result([]) → return
          불만족 & 잔여    : 스트림 중단 + rejected=후보이름 + reformulate → 재시도
                            (재검색어가 직전과 같으면 조기 종료 → 폴백)
  ```
- **만족 판정 = 선택 단계 재사용**: 별도 판정 LLM 없음. `parse_selection`이 유효 번호를 내면 만족.
- **스트리밍 엿보기**: 선택 응답을 스트리밍하며 첫 줄(`선택:`)만 버퍼링해 판정. 만족이면 나머지를
  토큰으로 흘리고, 불만족이면 스트림을 끊어 출력 토큰을 아낀다(현행 astream의 헤더 suppress 확장).
- **팔로업 답변**: 히스토리 + 질의로 텍스트 답변만 생성(검색·루프·카드 없음, `result([])`). 무상태
  멀티턴(ADR-0013)이라 서버엔 이전 상품 ID가 없으므로 카드 재표시는 범위 밖.
- **새 기본값(플래그 없음)**: 반복 루프가 기본 동작이 된다. prod는 재배포하지 않아 보호된다.
  `AGENT_MAX_ITERS=1`로 두면 사실상 단일 패스(분석기+1회)로 완화 가능.
- **런타임 배선**: `build_default_context`가 QueryAnalyzer를 만들어 RagRecommender에 주입.
- **.env**: `AGENT_MAX_ITERS`(기본 3). 기존 `RAG_TOP_K`·`AGENT_TOKEN_BUDGET`·`AGENT_HISTORY_TURNS` 유지.
- **원칙 유지**: 카드의 이름·가격·속성은 소스 하이드레이션에서 결정적으로(ADR-0001/0016). 루프는
  검색 리콜·정밀도만 높인다.

## Testing Decisions

- 좋은 테스트는 **외부 행동을 공개 인터페이스로** 검증한다(구현 내부 아님).
- `QueryAnalyzer`: ScriptedChatModel로 (a) 새 검색이면 키워드/시맨틱 파싱, (b) 팔로업이면 FOLLOWUP
  라우팅, (c) 히스토리 없으면 새 검색, (d) reformulate가 거부 후보를 반영, (e) JSON 파싱 실패 폴백.
  프리아트: 옛 QueryAnalyzer 테스트(git 히스토리).
- `RagRecommender` 루프(ScriptedChatModel + FakeRetriever, 결정적): (1) 1회차 만족 → 재시도 없음,
  (2) 1회차 없음 → reformulate → 2회차 만족, (3) N 소진 → 폴백, (4) 무진전 → 조기 종료,
  (5) 팔로업 → 검색 스킵·텍스트 답변·result([]), (6) 스트리밍: 재시도마다 status·최종만 token.
  프리아트: 기존 test_rag_recommender.py.
- `HybridRetriever.retrieve(keywords, semantic)`: 키워드별 검색 합집합·중복제거·설명 부착.
  프리아트: 기존 test_hybrid_retriever.py.

## Out of Scope

- 팔로업 시 이전 추천 카드 재표시(프론트가 이전 상품 ID를 보내는 계약 변경 필요).
- 별도 판정 LLM(Self-RAG) — 선택 단계 재사용으로 대체.
- 검색 점수 임계 기반 판정(임베딩 절대 점수는 신뢰도 낮음).
- 누적(union) 후보 — 이터레이션마다 교체.
- prod 배포·재빌드(시연 중, 건드리지 않음).

## Further Notes

- 이 변경은 **ADR-0015(비용 위해 QueryAnalyzer 제거)를 되돌린다** — 되돌리기 어렵고 트레이드오프가
  분명하므로 **ADR-0017**(반복 검색: 품질 위해 질의생성·루프 재도입, 채팅당 LLM 1→2~6) 1건 남긴다.
- 비용: 팔로업 2콜 · 새검색 만족 2콜 · 재시도 1회 4콜 · N=3 소진 6콜.
- 구현 후 **현행 단일 패스 vs 반복 루프 품질 비교**(대표 질의 세트로 A/B)를 수행한다.
