# PRD: 리랭크 스테이지 + 질의 정규화 (리콜 역전·검색 정밀도)

Status: ready-for-agent

## Problem Statement

넓은 질의가 좁은 하위질의보다 결과가 **적게** 나온다. "집게 있어?"는 코퍼스에 집게류가 14개
있는데도 1개(비이커집게)만 추천하고, 더 구체적인 "자석집게는?"은 4개(리트리버·반달집게·접시·
빠루망치)를 낸다 — 넓은 게 좁은 것의 상위집합이어야 하는데 역전이다. 게다가 자석집게 결과엔
방석·선반 같은 무관한 것도 섞인다.

원인은 검색이 아니라 **선택 단계**다(중간값 추적으로 확증). "집게"는 검색이 14개를 모두
후보로 올렸지만 선택 LLM이 13개를 버렸다. 현재 파이프라인엔 리랭커가 없고, 정밀도를 전적으로
선택 LLM의 통짜 판단에 맡기는데, 그 엄격도가 질의 표현에 따라 들쭉날쭉하다. 또한 분석기가 짧은
질의를 비대칭으로 확장한다("집게"→키워드 1개 vs "자석집게"→키워드 3개, 영어 포함).

## Solution

검색과 선택 사이에 **리랭크 스테이지**를 넣어, 후보마다 질의 적합도를 0~3으로 채점하고 그
**점수가 결과 집합을 주도**하게 한다(선택 LLM은 명백한 유형 불일치 제거와 근거 작성으로 강등).
동시에 **분석기의 질의 정규화를 공격적으로** 만든다(조사/의문 제거, KO/EN 항상 생성, 동의어·유형
확장). 리랭커가 정밀도를 뒤에서 받쳐주므로 앞단은 리콜 지향으로 넓게 뽑아도 안전하다. 결과적으로
"집게"에서 비이커집게·반달집게·리트리버가 일관되게 상위로 오고, 무관한 것은 점수 컷으로 걸러진다.

리랭커는 `Reranker` 딥모듈 인터페이스 뒤에 두어, 지금은 gpt-4o-mini 기반 `LLMReranker`로
구현하되 추후 로컬 크로스인코더·호스티드 API로 드롭인 교체가 가능하게 한다.

## User Stories

1. 사용자로서, "집게 있어?"에 비이커집게·반달집게·리트리버 등 집게류가 두루 나오길 원한다. 넓게 물었으니 넓게 받도록.
2. 사용자로서, 넓은 질의가 좁은 질의보다 결과가 적지 않길 원한다. 상식적인 포함관계가 지켜지도록.
3. 사용자로서, "자석집게"에 방석·선반 같은 무관한 것이 섞이지 않길 원한다. 관련 없는 잡음이 걸러지도록.
4. 사용자로서, 질의를 "집게"로 하든 "집게 있어?"로 하든 결과가 일관되길 원한다. 말끝의 조사·의문에 흔들리지 않도록.
5. 사용자로서, 한국어로 물어도 영어 상품명(Beaker Tong 등)이 잡히길 원한다. 표기 언어에 갇히지 않도록.
6. 사용자로서, "3000만원 이하" 같은 숫자 제약이 리랭크 후에도 존중되길 원한다. 하드 필터가 리랭크로 무너지지 않도록.
7. 사용자로서, 가격·순도가 비슷한 후보 중 더 적합한 게 상위로 오길 원한다. 리랭커가 레지스트리 값까지 보고 정렬하도록.
8. 사용자로서, 적합한 게 정말 없으면 엉뚱한 걸 억지로 보여주지 않길 원한다. 점수 컷 이하이면 재시도하거나 없다고 하도록.
9. 사용자로서, 첫 검색어가 빗나가도 시스템이 다른 검색어로 다시 찾아주길 원한다. 반복 루프가 유지되도록.
10. 운영자로서, 리랭크가 채팅 응답을 과하게 느리게 하지 않길 원한다. 질의당 콜이 폭증하지 않도록(전형 +1콜).
11. 운영자로서, 리랭크 비용이 감당 가능하길 원한다. 배치 1콜·질의당 ~$0.001 수준이도록.
12. 운영자로서, 좁아서 후보가 이미 적은 질의엔 리랭크 콜이 낭비되지 않길 원한다. 후보가 컷 이하이면 리랭크를 건너뛰도록.
13. 개발자로서, 리랭커를 나중에 크로스인코더/호스티드 API로 갈아끼울 수 있길 원한다. 인터페이스 뒤에 은닉되도록.
14. 개발자로서, 리랭커·선택·정규화 동작을 실제 LLM 없이 결정적으로 테스트할 수 있길 원한다. fake로 격리되도록.
15. 사용자로서, 리랭크가 붙어도 기존에 잘 되던 질의(원심분리기·실리카겔 등)가 나빠지지 않길 원한다. 회귀가 없도록.

## Implementation Decisions

### 파이프라인 (반복 루프, 최대 3회 유지)

```
analyze(정규화) → retrieve top-50 → rerank(0~3) → 점수≥2 & top-10 → select(유형필터+근거)
        ↑ 불만족(≥2 후보 전무 or 선택이 전부 유형불일치) → 검색어 재정식화 후 재시도
```

### 리랭커 (신규 딥모듈 `Reranker`)

- 인터페이스: `rerank(query, candidates) -> list[(candidate, score)]`. 후보마다 0~3 relevance 점수.
- 구현 `LLMReranker`: gpt-4o-mini에 후보 전체(이름 + 설명 + 레지스트리 값)를 **한 번의 배치 콜**로
  넣어 후보별 점수를 받는다(후보당 콜 아님). temp=0.
- **리랭커 주도**: 점수 임계(≥2)가 보여줄 후보 집합을 결정한다. 선택 LLM은 결과를 좌우하지 않는다.
- 교체성: 동일 인터페이스로 `CrossEncoderReranker`·호스티드 API를 드롭인 가능(이번엔 미구현).
- 프로토타입에서 확정된 점수 의미: `0` 무관 · `1` 약함 · `2` 적합 · `3` 매우 적합. 컷은 `≥2`.

### Funnel 폭·컷

- 검색 폭: 하이브리드 top-50(현행 RAG_TOP_K=20 → 리랭크 전용으로 확대. env로 조정).
- 리랭크 후: 점수 `≥2`만, 최대 10개 노출(env `RERANK_TOP_K=10`, `RERANK_MIN_SCORE=2`).
- **스킵 최적화**: 검색 후보 수가 노출 상한(10) 이하이면 자를 게 없으므로 리랭크 콜 생략.

### 리랭크 입력 (하이드레이션 전)

- 후보에 실어 보내는 값: 이름 + 설명(kg_description) + 레지스트리 4종(가격·순도·분자량·보관온도,
  kg_embedding 컬럼). 소스 하이드레이션은 리랭크 **승자 top-10에만** 적용(부하 최소).
- brand 등 소스 전체 사실은 리랭크 입력에 넣지 않는다(brand는 임베딩·설명에 이미 녹아 있음).

### 검색기 (`HybridRetriever` 수정)

- 검색 폭을 50으로. 각 후보에 레지스트리 값을 부착(리랭커가 숫자 판별을 하도록).
- 이를 위해 `EmbeddingStore`/`SemanticSearch`의 검색 결과 SELECT에 레지스트리 컬럼을 포함시킨다.
- 숫자 하드 필터(ADR-0018)는 검색 시점에 그대로 적용 — 리랭크는 필터 통과분에만 작동한다.

### 질의 정규화 (`QueryAnalyzer` 수정)

- ANALYZE_PROMPT 강화: (a) 조사·의문 제거("집게 있어?"→"집게"), (b) KO/EN 키워드 항상 생성,
  (c) 동의어·유형 확장(집게→tong/clamp/forceps/retriever). 인터페이스·반환 구조는 불변.
- 리랭커가 정밀도를 받치므로 리콜 지향으로 넓게 확장한다(비대칭 확장 해소).

### 선택 LLM (`RagRecommender` 내 역할 강등)

- 입력이 리랭크된 ≤10 후보로 축소된다. 역할: (a) 명백한 유형 불일치 제거(예: 플라스크 요청에
  합금), (b) 근거 작성. "선택: 없음"은 유지하되, 재시도 트리거에 "≥2 후보 전무"가 추가된다.
- 반복 루프는 유지하고 리랭크를 매 반복에 적용한다(재정식화 시 재검색→재리랭크).

### 배선·설정

- `RagRecommender(model, retriever, analyzer, reranker)`로 리랭커 주입(runtime 조립부).
- env: 검색 폭, `RERANK_TOP_K`, `RERANK_MIN_SCORE`, 리랭커 모델(기본 gpt-4o-mini).

### ADR

- 리랭크 스테이지 도입은 아키텍처 결정 → **ADR-0019** 작성(리콜/정밀 분리, 선택 LLM 강등,
  리랭커 주도, 교체 가능 인터페이스, LLM vs 크로스인코더 트레이드오프 기록).

## Testing Decisions

좋은 테스트는 **외부 동작만** 검증한다 — 실제 OpenAI 없이 fake 모델/컴포넌트로 결정적으로,
공개 인터페이스를 통해. 프롬프트 문자열·내부 파싱 구조가 아니라 "입력 질의/후보 → 결과·순위·
재시도" 같은 관찰 가능한 행동을 본다.

테스트 대상 4종(사용자 확인):

- **`Reranker`(LLMReranker)**: fake 모델이 캔드 점수를 반환 → 점수 파싱·정렬·임계 컷·스킵
  최적화(후보≤상한이면 콜 생략) 검증.
- **`RagRecommender`**: FakeReranker+FakeRetriever+fake 모델로 — 리랭커 주도(≥2만 노출),
  ≥2 전무 시 재시도, 선택 LLM의 유형 불일치 제거, 반복 루프 상호작용.
- **`QueryAnalyzer`**: fake 모델 구조 출력으로 — 조사/의문 제거·KO/EN·동의어 확장이 일관되게
  키워드에 반영되는지.
- **`HybridRetriever`**: 후보에 레지스트리 값이 실려오는지 + 검색 폭 50 + 하드 필터 동시 적용.

프라이어아트: `test_rag_recommender.py`(fake 모델·리트리버 패턴), `test_hybrid_retriever.py`,
`test_query_analyzer.py`, `test_embedding_store.py`(실 pgvector·kg_embedding_test).

## Out of Scope

- 로컬 크로스인코더/호스티드 리랭크 API 구현(인터페이스만 열어두고 이번엔 LLMReranker만).
- 제조사/판매사 회사명 표시(코드→회사명 매핑이 소스 덤프에 없음 — 고객사 실데이터 연동 별건).
- 사전 기반 동의어/철자 정규화 모듈(LLM 확장으로 대체).
- 61만 전면 적재·HNSW 실빌드(이미 코드에 있음, 별도 실행 항목).
- 리랭크 품질의 판정관 A/B 스코어링(후속 평가로, 본 PRD는 기능 도입까지).

## Further Notes

- **콜/비용**: 질의당 LLM chat 콜이 전형 +1(리랭크 배치 1콜), 최악(3반복) +3. 리랭크 ~$0.001/콜,
  질의당 총 ~$0.003–0.005. 신규 인프라 0(기존 OpenAI 재사용).
- **진단 근거**: "집게" 검색 후보 20개에 집게류 14개 전부 포함됐으나 최종 1개만 선택됨(선택 LLM
  병목 확증). "자석집게"는 후보가 자석교반기·잡음에 지배됨(정규화·리랭크로 개선 대상).
- 후속 평가는 기존 판정관 방식(절대 0~3 + pairwise)으로 리랭크 전/후 비교 권장(`rag-ablation` 참고).
