# 숫자 범위 하드 필터 — 임베딩이 못 하는 것만 필터로

임베딩 검색(ADR-0012/0015)은 유형·용도·재질·**브랜드** 같은 어휘/의미 신호엔 강하다(실측:
"ATAGO 굴절계"→ATAGO만, "Nalgene 병"→Nalgene만). 하지만 **숫자 범위 비교**엔 무력하다.
"3000만원 이하"는 가격이 색인에 없어 필터되지 않고, 숫자 "3000만"이 어휘로 섞여 "3000 mL"에
오매칭된다. "순도 99% 이상", "분자량 100 이하", "냉장(2~8℃) 보관"도 마찬가지다.

## 결정

**하드 필터는 임베딩이 못 하는 "숫자 범위 비교"에만 붙인다. 범주형은 임베딩이 처리한다.**

- **메타데이터 필터 벡터 검색**: 적재 시 숫자 속성(가격·순도·분자량·보관온도)을 kg_embedding
  숫자 컬럼(min/max)에 색인하고, 검색 SQL에 `WHERE ... ORDER BY embedding<->q`로 **필터+시맨틱
  동시** 적용. 사후 하이드레이션 필터가 아니라 "필터된 집합에서 시맨틱 순위"여야 정확하다.
- **선언적 레지스트리**(FILTER_SPEC): 각 필터를 (이름, doc→(min,max) 추출)로 선언 → 적재·색인·질의·
  제약추출이 이 위에서 일반화. 컬럼 추가 = 1줄.
- **저장 = 타입 min/max 컬럼**: 원본이 범위면 그대로("2-8C"→(2,8)), point면 min=max(분자량 71.08),
  가격은 변형 io_price의 min/max. JSONB 아님(숫자 비교·인덱스 깔끔).
- **정규화**: 질의 시 = 분석기 LLM(레지스트리 스키마 주입)이 제약 추출("3000만원 이하"→price_max);
  적재 시 = 결정적 파서(텍스트→숫자, storage는 소어휘 매핑). storage 온도매핑(냉장=2~8·냉동≤-20·
  실온=15~25)은 양쪽이 같은 정의 공유 → "냉장"과 "2~8도"가 동일 작동.
- **매칭 = 겹침(overlap)**: 이하(≤X)→`col_min≤X`; 이상(≥X)→`col_max≥X`; 범위[A,B]→`col_min≤B AND
  col_max≥A`. 값 NULL이면 그 필터에서 제외.
- **하드·필터-범인 감지**: 필터는 자동 완화 없음, 루프 내내 유지(재정식화는 검색어만). 필터 걸고
  0건이면 **필터 빼고 1회 더 검색** → 결과 있으면 = 필터가 범인 → 검색어 재시도 대신 구체적 안내.

## 이유

- **역할 분리**: 임베딩은 의미/어휘(브랜드·재질·유형)를 잘한다(실측). 못 하는 건 숫자 비교뿐이니
  거기만 하드 필터를 붙이면 최소 변경으로 최대 효과. 브랜드 별칭맵·범주 통제어휘 불필요.
- **정확성**: 하드 제약(예산)은 "시맨틱 상위에서 거르기"가 아니라 "필터된 집합에서 순위"여야 한다.
  SQL `WHERE ... ORDER BY embedding<->`가 정확히 이걸 한다.

## 대안 (기각)

- **범주형도 하드 필터**(브랜드·보관·단위): 임베딩이 이미 처리(실측) → 불필요·정규화 비용만. 브랜드
  별칭맵(SIAL→Sigma)은 후속.
- **사후 하이드레이션 필터**: 채팅당 소스 히트 K배↑ + 시맨틱 상위에만 필터 적용(과소).
- **필터 자동 완화**: 예산/스펙 하드 제약을 어기게 됨. 하드 유지, 0건은 정직히 안내.

## 결과 / 트레이드오프

- kg_embedding에 필터 컬럼(price·purity·molecular_weight·storage_temp의 _min/_max). C의 "소스 복제
  안 함"과 타협하나, 전체가 아닌 **필터 몇 컬럼**만 색인하고 content_hash로 sync 갱신된다.
- **신규 컬럼은 기존 행에 NULL** → 배포 시 **재적재로 채워야** 함(`embed_products --reset` 또는 backfill).
- v1 = 가격·순도·분자량·보관온도(파서 다양성 대표). 희소(녹는점·인화점·밀도, 1~19%)·범주형은 후속으로
  레지스트리에 1줄씩. 브랜드 완벽 매칭도 후속.
