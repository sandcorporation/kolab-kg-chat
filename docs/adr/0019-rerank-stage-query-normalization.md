# 리랭크 스테이지 — 검색과 선택 사이 리랭커 주도, 선택 LLM 강등

검색 리콜은 충분한데(진단: "집게" 14개가 모두 후보에 오름) 최종 추천이 빈약했다(1개). 병목은
검색이 아니라 **선택 LLM**이었다 — 정밀도를 통짜 판단에 맡기니 질의 표현에 따라 엄격도가
들쭉날쭉해, 넓은 질의("집게")가 좁은 하위질의("자석집게")보다 결과가 적어지는 역전이 났다.
또 분석기가 짧은 질의를 비대칭으로 확장해("집게"→키워드 1개) 리콜도 불안정했다.

## 결정

**검색과 선택 사이에 리랭크 스테이지를 넣고, 리랭커 점수가 결과 집합을 주도한다. 선택 LLM은
유형 불일치 제거·근거 작성으로 강등한다. 분석기는 리콜 지향으로 공격적으로 정규화한다.**

```
analyze(정규화) → retrieve top-50 → rerank(0~3) → 점수≥2 & top-10 → select(유형필터+근거)
        ↑ 불만족(≥2 후보 전무 or 선택이 전부 유형불일치) → 검색어 재정식화 후 재시도
```

- **리랭커 주도**: 후보마다 질의 적합도 0~3을 매겨 임계(≥2) 이상만 점수순 top-K로 남긴다. 선택
  LLM은 이 통과분에서만 고르므로 결과 집합을 좌우하지 못한다(랭킹은 리랭커가 안정적으로).
- **리랭커 = 교체 가능 딥모듈**: `rerank(query, candidates)` 인터페이스. 1차 구현 `LLMReranker`
  (gpt-4o-mini, 후보 전체 **1회 배치 콜**로 채점 — 후보당 콜 아님). 동일 인터페이스로
  로컬 크로스인코더·호스티드 API 드롭인 교체.
- **하이드레이션 전 리랭크**: 입력 = 이름 + 설명(kg_description) + 레지스트리 값(가격·순도·분자량·
  보관온도, 이미 kg_embedding 컬럼). 승자 top-10만 소스 하이드레이션(부하 최소). brand는 임베딩·
  설명에 이미 녹아 있어 별도 안 넣음.
- **넓게 검색 → 리랭크로 컷**: 검색 폭을 50으로 확대(리콜↑), 리랭커가 정밀도 복구. 분석기도 KO/EN·
  동의어·유형을 넉넉히 확장하고 말미 군더더기·의문은 뗀다("집게 있어?"→"집게").
- **스킵·재시도**: 후보 수가 top-K 이하이면 자를 게 없어 리랭크 콜 생략. 통과 후보 0이면 선택을
  건너뛰고 반복 루프가 재검색(기존 "선택: 없음" 경로 재사용).

## 이유

- **정밀도의 안정성**: 후보별 독립 0~3 채점은 "통짜 선택"보다 질의 표현에 덜 흔들린다 → 리콜 역전
  해소. 리콜은 검색·정규화가, 정밀은 리랭커가 — 책임 분리.
- **최소 마찰**: LLM 리랭커는 기존 OpenAI 경로 재사용(신규 인프라 0), 질의당 +1 배치 콜(~$0.001).
  인터페이스 뒤에 은닉해 나중에 로컬/호스티드로 무중단 교체.

## 대안 (기각)

- **리랭커 보완(재정렬만, 선택 LLM이 최종 결정)**: 같은 엄격한 LLM이 여전히 버려 "집게→1"이 안
  고쳐진다. 병목을 안 건드림.
- **로컬 크로스인코더 우선**: 결정적·$0이나 이미지 +2~3GB·CPU 1~3s 지연. 인터페이스만 열어두고 후속.
- **하이드레이션 후 full-facts 리랭크**: 정확하나 top-50 전량 하이드레이션(부하 10×·~$0.004-0.015/q).
- **사전 기반 동의어 정규화**: 결정적이나 도메인 사전 유지보수 부담 → LLM 확장으로 대체.

## 결과 / 트레이드오프

- 질의당 LLM 콜 전형 +1(리랭크 배치 1콜), 최악(3반복) +3. 비용 질의당 ~$0.003-0.005, 지연은
  리랭크 배치 콜 1회분 추가. 후보≤top_k 스킵으로 좁은 질의는 콜 없음.
- 검색 폭 기본 20→50, env(`RAG_TOP_K`·`RERANK_TOP_K`·`RERANK_MIN_SCORE`)로 조정. 선택 LLM은
  이제 리랭크 통과분(≤10)만 보므로 컨텍스트·토큰도 절감.
- 리랭크 품질의 판정관 A/B 스코어링은 후속 평가(`rag-ablation` 방식)로 남긴다.
