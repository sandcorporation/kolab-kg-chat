# 임베딩·semantic_search를 운영 읽기 경로에 재도입한다

ADR-0010은 임베딩·시맨틱 검색을 제거했다(당시엔 읽기 경로가 쓰지 않았고 비용·복잡도가 순손실이라 판단). 그 결정을 실증으로 검증하려 리트리벌 스택 ablation을 돌렸고([RESULTS](../../.scratch/retrieval-quality-eval/RESULTS.md)), **임베딩이 실제로 추천 품질을 높인다**는 근거가 나왔다. 이 ADR은 **ADR-0010을 대체(supersede)** 한다.

## 근거 (실험)

- 실 카탈로그에서 큐레이션한 코퍼스 250 · 계층화 쿼리 31 · 절대 루브릭 채점(0~3, gpt-4o) + 블라인드 쌍별 심사(순서스왑). 에이전트 temp=0, 모든 LLM 산출 캐시.
- **config4(그래프 도구 + `semantic_search` 임베딩)가 최고** — 절대 2.42/3, 쌍별 승률 0.363.
- 특히 **semantic(서술형·한/영·유의어) 질의에서 2.9/3** — 임베딩 없는 config1(SQL) 2.0, config2(structured만) 2.5 대비 뚜렷하게 앞섰다. 임베딩 없는 SQL 베이스라인은 최하위권(1.68).
- (반대 방향 검증) 별도로 시도한 하이브리드 융합+리랭킹(config5)은 오히려 최하(1.65)였다 — 단순 융합이 답은 아니며, **임베딩을 tool로 쓰는 에이전트 경로(config4)가 실질 승자**임을 재확인.

## 결정

- **운영 Recommendation Agent(ADR-0011)에 `semantic_search` 도구를 추가**하고, **쓰기 경로(적재)에서 Product 텍스트(name + 속성값)를 임베딩**한다(`text-embedding-3-small`, pgvector). ADR-0003의 pgvector 축이 다시 활성화된다.
- 실험 하네스의 lean 구현(임베딩 저장·최근접 검색)을 운영으로 승격하되, **임베딩 캐시(content-hash)·모델 버전 태깅**을 유지한다.
- **부분 부활**: ADR-0009(임베딩 대상·모델)를 lean 형태로 되살린다 — 상품 텍스트만 임베딩하고, Application/Condition 노드 임베딩은 범위 밖으로 둔다.
- **vision은 재도입하지 않는다** — 같은 실험에서 config3(+vision)은 뚜렷한 값을 못 보였고(이미지가 외부 CDN·상품사진), ADR-0005는 저순위로 유지.

## 이유

- semantic 질의(서술형·유의어·한영 미스매치)는 키워드(이름 CONTAINS)나 속성 필터로 못 잡는데 임베딩이 그 간극을 메운다 — 실험이 정량 확인.
- `text-embedding-3-small`은 저렴해(전 카탈로그도 수 달러 규모) vision과 달리 비용 부담이 작다. ADR-0010이 우려한 "순비용"의 전제(읽기 경로 미사용)가 뒤집혔다.
- pgvector는 ADR-0010이 db 이미지에 남겨둬 재도입 장벽이 낮다.

## 결과

- 적재 비용·복잡도가 다시 는다(임베딩 생성·모델 교체 시 재임베딩·pgvector 조회). content-hash 임베딩 캐시로 재적재 비용을 억제한다.
- ADR-0010 무효, ADR-0009 lean 부활, ADR-0003의 pgvector 축 재활성.
- **소표본(쿼리 31)·심사 1모델/1런** 실험이라 결정적이기보다 방향성이다 — 운영 배포 후 추천 품질을 모니터링하고, 반증 시 재검토한다.
- 후속 실험 여지: 하이브리드+리랭킹(config5)은 이번엔 실패했으나 후보 풀 확대·속성 융합·리랭커 개선으로 재도전할 수 있다(별개 실험).
