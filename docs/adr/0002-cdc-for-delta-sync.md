# 소스 DB의 변경은 CDC로 구독한다

플랜은 초기 전체 적재 후, 이후 delta(생성·수정·삭제)마다 개별 row를 그래프에 점진 반영해야 한다. 소스 DB는 우리 소유가 아닌 kolabshop의 운영 DB이며, 약 2026-06-29에 접근이 제공될 예정이다.

소스 DB의 트랜잭션 로그(MySQL binlog / Postgres WAL)를 **CDC(Change Data Capture)** 로 구독해 row 변경을 실시간 포착한다. 타임스탬프 폴링, DB 트리거, 앱 레벨 이벤트는 기각한다.

이유: CDC는 실시간이고, **하드 삭제까지 포착**하며, 운영 DB에 거의 부하를 주지 않는다. 폴링은 삭제를 놓치고 신뢰할 `updated_at`을 요구한다. 트리거는 남의 운영 DB를 수정해야 한다. 앱 이벤트는 쇼핑몰 앱의 협조가 필요하다. 사용자의 "DB를 구독하는 서비스" 직감과도 일치한다.

## 전제 및 대비책

- binlog/WAL 또는 읽기 복제본(replica) 수준의 접근에 의존한다. DB 도착 시(~2026-06-29) 확정한다.
- 만약 읽기 전용 SELECT만 허용되면, **타임스탬프 폴링 + 주기적 전체 대조(reconciliation)** 로 삭제를 보정하는 방식으로 후퇴한다.
