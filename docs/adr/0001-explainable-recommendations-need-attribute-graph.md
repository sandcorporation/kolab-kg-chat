# 추천은 근거를 댈 수 있어야 하므로, 그래프는 정규화된 기능 속성을 저장한다

이 챗봇은 적합성(fitness-for-purpose) 추천 질의("~~ 실험을 ~~ 환경에서 ~~를 쓰며 기획 중인데, 무엇을 사야 할까?")에 답하며, **근거 설명은 필수 요구사항**이다 — 추천된 모든 상품은 *어떤 요구조건을* 충족하는지 인용할 수 있어야 한다.

따라서 그래프 노드에 정규화된, 이름 붙은 **기능 속성(Functional Attribute)**(재질, 온도 범위, 멸균 등급, 용량, 호환 규격 등)을 저장하고, 응용(Application)·조건(Condition)의 요구조건을 그 속성과 매칭하여 검색한다(GraphRAG). 상품 설명에 대한 순수 벡터 유사도 RAG는 기각했다: 불투명한 유사도로 순위를 매길 뿐, 어떤 요구조건을 충족하는지 말하지 못하기 때문이다.

실험→요구조건 매핑 자체는 큐레이션된 도메인 온톨로지에 박아 넣는 대신 **LLM이 질의 시점에** 수행한다(요구조건을 명시적으로 출력해 인용 가능하게 한다). 특정 매핑이 검증되면 이후 그래프로 승격한다.

## 결과

- 어렵고 가치 있는 엔지니어링은 브랜드/카테고리 그래프 구축이 아니라, **지저분한 텍스트와 OCR로 읽은 이미지에서 구조적 속성을 추출하는 일**이다.
- 추천은 감사(audit) 가능하지만, LLM이 질의 시점에 내놓는 요구조건 추론이 사용자에게 드러나고 교정 가능하다는 점에 의존한다.
