# CDC 반영은 Product 단위 코얼레싱 + 현재 상태 재조립으로 한다

CDC는 row 단위 변경을 흘린다(ADR-0002). 그러나 한 Product는 여러 row·테이블로 조립된다(질문 10, CONTEXT의 Product). 비전 LLM 추출(ADR-0005)은 비싸다. delta 반영 지연은 느슨해도 된다(확정).

## 결정

- **큐 작업 단위 = Product ID**, 중복제거/디바운스로 **코얼레싱**. 한 Product에 변경이 버스트로 와도 **한 번만** 재처리. 지연 허용이 느슨하므로 디바운스 창을 넉넉히(수십초~분), 워커는 적게.
- 워커는 CDC 이벤트를 **재생하지 않고** 소스의 **현재 상태**를 다시 읽어 Product를 재조립한다 → 멱등(last-write-wins).
- **content-hash 게이팅**: 재조립한 Product 문서(텍스트·이미지 참조)의 해시를 저장해, 바뀐 부분만 재추출하고 비전 LLM 재호출을 최소화한다.
- **upsert는 소스 PK 기준** 멱등(중복 노드/엣지 생성 금지). 초기 전체 적재도 같은 워커 경로 — 모든 Product를 한 번 enqueue.

## 이유

이벤트 재생은 비전 LLM 비용을 폭증시키고(옵션 5개 동시 변경 = 재추출 5회) 순서·중복 문제를 떠안는다. "변경을 신호로 깨우고 현재 상태를 멱등하게 반영"이 CDC 파이프라인의 정석이며, 비전 LLM이 비싼 우리 비용 구조에 결정적이다.

## 결과

- delta가 즉시 반영되지 않는다(느슨한 지연 — 허용됨).
- 큐 기술은 Django-async에 맞는 taskiq 정도(부차적 선택).
- content-hash 스냅샷은 provenance(ADR-0004)와 함께 "이 속성이 어느 스냅샷에서 왔는가"를 추적한다.
- 쓰기 경로 워커는 챗 읽기 경로와 분리된다(ADR-0007).
