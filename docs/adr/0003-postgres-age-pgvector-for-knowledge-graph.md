# Knowledge Graph는 Postgres + Apache AGE + pgvector에 저장한다

> **개정됨(ADR-0010 → ADR-0012).** ADR-0010이 의미 유사도(pgvector) 축을 뺐으나, 실험 근거로 ADR-0012가 임베딩·semantic_search를 재도입하며 **pgvector 축이 다시 활성화**된다. 질의 모델은 다시 삼중: AGE 순회 + SQL/키워드·속성 필터 + pgvector 의미 유사도.

이 프로젝트는 한 저장소에서 세 가지를 동시에 요구한다: (1) 정규화된 속성에 대한 필터·숫자 범위 질의(ADR-0001), (2) **가변 깊이의 명시적 엣지 순회**(호환·부속·소모품 의존 체인 — "X를 쓰려면 결국 무엇이 다 필요한가"), (3) 의미 유사도(GraphRAG의 "비슷한 응용/환경" 회수). 더해 langgraph가 이미 Postgres에 상태를 영속화한다.

Knowledge Graph를 **단일 Postgres 인스턴스**에 저장하되, **Apache AGE(openCypher)** 로 그래프 순회를, **pgvector** 로 의미 유사도를, **순수 SQL** 로 속성 필터를 수행한다. 에이전트의 검색이 이 셋을 조합한다.

## 기각한 대안

- **Neo4j(네이티브 그래프+벡터, langchain 1급 통합)** — 본래 가장 자연스러운 선택이었으나, 두 번째 데이터스토어 도입을 피하고 Postgres로 일원화하기 위해(팀/운영 선호) 기각.
- **순수 관계형 + 재귀 CTE** — 호환·의존 체인 같은 *가변 깊이 명시적 순회*가 핵심 요구사항이라 기각(CTE로는 고통스럽고 느려짐). 유사도 기반 "비슷함"은 순회가 아니라 pgvector로 푼다는 점은 유효하나, 그것만으로는 부족.

## 결과

- **이중(삼중) 질의 모델**: 순회는 Cypher(AGE), 유사도는 pgvector, 속성 필터는 SQL. 에이전트의 검색 계층이 이를 합성해야 한다.
- AGE는 상대적으로 니치한 확장이다 — 성숙도·운영 리스크를 모니터링한다.
- 같은 Postgres 인스턴스가 KG 스키마와 langgraph 체크포인터/메모리를 **분리된 스키마**로 함께 담는다. KG를 langgraph의 `PostgresStore`(KV 메모리)에 넣지 않는다.
